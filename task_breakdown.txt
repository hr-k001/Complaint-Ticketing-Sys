Complaint Ticketing System - Team Task Breakdown
===============================================

How the full project should work
--------------------------------
We are building a complaint / ticketing system with 3 roles:
- user
- agent
- admin

High-level working
------------------
1. A user registers and logs in.
2. The user creates a complaint/ticket.
3. The system assigns priority, category, status, and SLA due date.
4. The user sees only their own tickets in the user dashboard.
5. Agents log in and see only tickets assigned to them in the agent dashboard.
6. Agents can update ticket status and add comments on the tickets they manage.
7. Admin logs in and sees all tickets, all users, assigned agents, escalations, and dashboard reports.
8. If a ticket crosses the SLA due date and is still unresolved, escalation logic marks it as escalated.
9. Users should mainly track tickets using ticket number, while the backend internally uses ticket id.

How accounts should work in the final UI
----------------------------------------
In the final project, not everyone should be able to become an admin or agent from the public UI.

Recommended final flow:
- public registration should create only normal user accounts
- agent and admin accounts should be created only by admin or seeded manually in the database
- later, if needed, we can add an admin-only screen to create agent/admin accounts

Recommended rule:
- register page -> only user role
- admin/agent role creation -> admin-controlled only

For now in backend testing, role can still be passed during registration because the system is under development.
and i have created a create_user.py script to safely create agent/admin accounts directly when the whole project is completed

Agent dashboard

Admin dashboard
- all tickets

Important design note
---------------------
- users should mainly see ticket_number, not raw ticket_id
- agents may also mostly work with ticket_number in UI
- admin can see both ticket_number and ticket_id if needed for internal management
- frontend can keep internal ids hidden while still using them in API calls
